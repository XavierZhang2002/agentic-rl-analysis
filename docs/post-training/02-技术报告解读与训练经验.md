## 一、DeepSeek-R1 — 纯 RL 推理涌现

> **报告**: *DeepSeek-R1: Incentivizing Reasoning Capability in LLMs via Reinforcement Learning*
> **arXiv**: [2501.12948](https://arxiv.org/abs/2501.12948) (2025.01)
> **核心主张**: 纯 RL 可以让推理能力从零涌现，不需要人工标注的推理轨迹

### 1.1 动机分析

DeepSeek-R1 面对的核心问题是：**如何让模型学会复杂的多步推理（如竞赛级数学），而不需要昂贵的人工推理链标注？**

传统思路（SFT on CoT data）受限于标注员的推理能力和标注成本。DeepSeek 提出了一个大胆假设：基座模型在预训练中已经蕴含了推理的"种子"，RL 的作用是**选择性地激活和强化这些种子**，而非从零教授推理。

### 1.2 Pipeline：四阶段

```
DeepSeek-V3 Base (671B MoE, 37B 激活参数)
    │
    ▼ 阶段 1: Cold Start SFT
    │   数据: 数千条长 CoT 样本（人工标注或少量蒸馏）
    │   目的: 学会 <think>...</think> 格式，避免 R1-Zero 的格式混乱
    │
    ▼ 阶段 2: Reasoning RL ← 核心阶段
    │   算法: GRPO (无 Critic, 无 RM)
    │   奖励: 基于规则 (数学答案匹配 + 格式检查)
    │   发现: "aha moment" — 模型自发学会反思和纠错
    │
    ▼ 阶段 3: Rejection Sampling + SFT
    │   从阶段 2 模型采样 ~800K 高质量回答 → 重新 SFT
    │   目的: 巩固推理能力 + 恢复通用能力
    │
    ▼ 阶段 4: All-Scenario RL
    │   多种任务上做最终对齐 (有用性 + 安全性)
    │
    ▼
DeepSeek-R1 (AIME 2024: 79.8% pass@1)
```

### 1.3 R1-Zero：纯 RL 的涌现与局限

DeepSeek 做了一个极端实验——**R1-Zero**：跳过所有 SFT，直接在基座模型上做 GRPO。

**涌现行为**（报告 Figure 3-4）:

| 行为 | 描述 | 意义 |
|------|------|------|
| "aha moment" | 模型自发学会 "Wait, let me reconsider..." 并纠正推理错误 | 自我反思能力从 RL 中自然涌现，不是被教会的 |
| 思维链自然延长 | 困难问题上自动生成更长的推理过程 | 模型学会了按需分配"思考预算" |
| 自我验证 | 学会回头检查计算结果 | 类似人类的元认知能力 |

**暴露的问题**：语言混杂（多种语言混用）、可读性差、格式不稳定。这些问题说明**纯 RL 能涌现推理能力，但需要 SFT 来提供基本的格式和可读性约束**。

### 1.4 失败尝试（报告 Section 3.4 — 极有价值的负面结果）

| 尝试 | 结果 | 报告中的分析 |
|------|------|------------|
| **Process Reward Model (PRM)** | 效果不如 Outcome RM | PRM 难以准确标注中间步骤的正确性，容易被 reward hack |
| **Monte Carlo Tree Search (MCTS)** | 在 LLM 场景不 work | 搜索空间太大、value 估计不准、token-level 搜索不匹配推理粒度 |
| **直接基座 RL（R1-Zero）** | 能力涌现但不可用 | 需要 SFT 提供格式约束 |

### 1.5 GRPO 的选择理由

报告给出的技术论证：
- 对 671B MoE 模型，训练同等规模的 Critic 网络**显存不可承受**（节省 ~25%+ 显存）
- 数学/代码任务有确定答案，基于规则的奖励 + Group Relative 优势已经足够
- 避免 Critic 在长 CoT 推理中的累积误差

## 二、DeepSeek-V3 — 高效后训练与蒸馏

> **报告**: *DeepSeek-V3 Technical Report*
> **arXiv**: [2412.19437](https://arxiv.org/abs/2412.19437) (2024.12)
> **核心主张**: 通过 R1 蒸馏 + Self-Rewarding，以极低算力完成高质量后训练

### 2.1 动机分析

DeepSeek-V3 面对的问题是：**如何在已有 R1 推理能力的前提下，高效地将这种能力整合到通用模型中？** 从头做 RL 太贵，但直接 SFT on R1 数据又会损失通用性。

### 2.2 关键数据

| 指标 | 数值 | 意义 |
|------|------|------|
| SFT 数据量 | ~1.5M instances | 大规模蒸馏 + 合成 |
| Post-Training 总算力 | **仅 5K H800 GPU hours** | 极高效率 |
| MTP 接受率 | 85-90% | Multi-Token Prediction 加速推理 |

### 2.3 R1 蒸馏 Pipeline

```
阶段 1: 用 R1 生成推理数据
阶段 2: 双轨 SFT
   ├─ 推理任务: 用 R1 生成的长 CoT 回答
   └─ 通用任务: 用标准格式回答
阶段 3: RL 微调 (使用 Self-Rewarding)
阶段 4: Rejection Sampling
```

**Self-Rewarding** 的核心思想：模型自己给自己的输出打分，用于构建偏好数据，避免训练独立 RM。这是 Constitutional AI 思路在推理模型上的延伸。

### 2.4 关键技术洞察

报告揭示了一个重要的工程权衡：**蒸馏 vs 从头 RL**

| 维度 | R1 蒸馏 | 从头 RL |
|------|---------|---------|
| GPU 成本 | ~5K H800 hours | 估计 10× 以上 |
| 推理能力 | 接近 R1 水平 | 可能超越 R1 |
| 通用能力 | 需要双轨 SFT 保持 | 各阶段 RL 分别优化 |
| 风险 | 低（SFT 是确定性过程） | 高（RL 不稳定） |

## 三、Qwen3 — Thinking Mode Fusion

> **报告**: *Qwen3 Technical Report*
> **arXiv**: [2505.09388](https://arxiv.org/abs/2505.09388) (2025.05)
> **核心主张**: 在单一模型中统一 thinking 和 non-thinking 模式，用极少量数据实现大幅推理提升
>
> **相关报告**:
> - Qwen2.5 Technical Report: [arXiv:2412.15115](https://arxiv.org/abs/2412.15115)
> - GSPO (Qwen3 的核心 RL 算法): [arXiv:2507.18071](https://arxiv.org/abs/2507.18071)

### 3.1 动机分析

Qwen3 面对两个核心挑战：

1. **Thinking 模式的用户体验问题**: 所有问题都用长 CoT 回答（像 DeepSeek-R1）会导致简单问题也输出大段思考过程，用户体验差且推理成本高
2. **数据效率问题**: 能否用最少的数据和 RL 步数，释放基座模型已有的推理潜力？

### 3.2 Pipeline：四阶段

```
Qwen3 Base (Dense: 0.6B-32B, MoE: 30B-235B)
    │
    ▼ 阶段 1: Long-CoT Cold Start SFT
    │   少量高质量长 CoT 数据，学会 <think>...</think> 格式
    │
    ▼ 阶段 2: Reasoning RL
    │   算法: GRPO
    │   数据: 仅 3,995 个 query-verifier 对！
    │   结果: AIME 2024 从 70.1 → 85.1 (仅 170 步 RL)
    │
    ▼ 阶段 3: Thinking Mode Fusion SFT ← 核心创新
    │   混合 thinking 模式 + non-thinking 模式数据
    │   让模型学会根据问题难度自动选择是否"思考"
    │
    ▼ 阶段 4: General RL
    │   全场景对齐，使用 GSPO 算法
    │
    ▼
Qwen3 Chat (thinking/non-thinking 自由切换)
```

### 3.3 关键技术洞察

**发现 1：极高的数据效率**

仅 **3,995 个 query-verifier 对 + 170 步 RL** 就将 AIME 2024 从 70.1 提升到 85.1（+15 分）。这强力佐证了"RL 释放已有能力"而非"RL 教授新能力"的观点——基座模型已经蕴含了推理能力。

**发现 2：Thinking Mode Fusion 的实现**

报告描述了 Thinking Mode Fusion SFT 的关键细节：
- 混合 thinking 模式数据（复杂问题 + 长 CoT）和 non-thinking 模式数据（简单问题 + 直接回答）
- 模型学会通过问题特征判断是否需要启动思考
- 涌现出"Thinking Budget"能力——根据问题难度自适应分配思考长度

**发现 3：强到弱蒸馏**

用大模型的推理能力蒸馏到小模型，GPU 成本仅为从头 RL 训练的 $1/10$。这为资源受限的团队提供了实用路径。

### 3.4 Qwen2.5 的对比参考

> **arXiv**: [2412.15115](https://arxiv.org/abs/2412.15115)

Qwen2.5 采用了更传统的四阶段流水线（SFT → Offline DPO → Online GRPO → 长上下文微调）。相比之下，Qwen3 的核心变化是：

| 维度 | Qwen2.5 | Qwen3 |
|------|---------|-------|
| RL 算法 | GRPO | GRPO → GSPO |
| DPO 阶段 | 核心阶段 (~150K 偏好对) | 不再使用 |
| 推理模式 | 无 thinking mode | Thinking Mode Fusion |
| RM 架构 | 6 维 RM (有用性/安全性/正确性/格式/完整性/风格) | 继承 + 简化 |

Qwen2.5 的**6 维 RM** 值得关注——多维 RM 允许在 RL 阶段对不同维度设置不同权重（如安全性高权重、格式低权重），避免了单一分数 RM 的模糊性。

## 四、Kimi K1.5 / K2 — Agentic Post-Training

> **K2 报告**: *Kimi K2: Open Agentic Intelligence*
> **K2 arXiv**: [2507.20534](https://arxiv.org/abs/2507.20534) (2025.07)
>
> **K1.5 报告**: *Kimi k1.5: Scaling Reinforcement Learning with LLMs*
> **K1.5 arXiv**: [2501.12599](https://arxiv.org/abs/2501.12599) (2025.01)
>
> **核心主张**: 大规模 Agentic 训练数据合成 + 自批评奖励机制

### 4.1 动机分析

Kimi 团队面对的核心问题与 DeepSeek/Qwen 不同：**如何让模型学会使用工具和在真实环境中完成任务？** 纯数学推理的 RLVR 范式无法直接应用于 Agentic 任务，因为"工具调用是否正确"很难用简单规则判定。

### 4.2 K1.5：推理 Scaling 的简约路线

K1.5 报告（arXiv:2501.12599）的核心发现：

- 使用长上下文扩展 + 改进策略优化实现 SOTA 推理（AIME 77.5, MATH500 96.2）
- **无需 MCTS、价值函数或过程奖励模型**——与 DeepSeek-R1 的结论一致
- 提出 **long2short** 方法：用长 CoT 技术提升短 CoT 模型性能

### 4.3 K2：Agentic Post-Training

| 指标 | 数值 |
|------|------|
| 总参数 | 1.04T (MoE) |
| 激活参数 | ~32B |
| RL 算法 | GRPO 变体 |

**Agentic 数据合成**: K2 最大的创新是大规模 Agentic 训练数据的合成：

- 集成 **3000+ 真实 MCP (Model Context Protocol) 工具**
- 让模型在**真实工具环境**中交互，收集轨迹（而非模板生成）
- 自动化流程不断扩展工具覆盖面

**Self-Critique Rubric Rewards**: K2 没有训练独立的 RM，而是让模型自己根据预定义的评分维度（rubric）对回答进行批评：

1. 模型生成回答
2. 模型根据 rubric **自我批评**
3. 自我批评结果 → 奖励信号

优势：评分标准通过 rubric 灵活调整，无需重新训练 RM。

**MuonClip 优化器**: K2 使用自研的 MuonClip 优化器，在 15.5T token 的预训练中实现**零 loss spike**——这对 MoE 模型训练至关重要（MoE 的训练不稳定性是公认难题）。

## 五、Seed1.5-Thinking — DAPO/VAPO 工业验证

> **报告**: *Seed1.5-Thinking: Advancing Superb Reasoning Models with Reinforcement Learning*
> **arXiv**: [2504.13914](https://arxiv.org/abs/2504.13914) (2025.04)
> **核心主张**: VAPO 在工业规模上稳定优于 DAPO；RFT 会伤害后续 RL

### 5.1 动机分析

Seed1.5-Thinking 的独特价值不在于提出新算法，而在于**在工业规模上系统验证了 DAPO 和 VAPO 的相对表现**。这是目前唯一在 200B MoE 模型上对两种 RLVR 路线进行 head-to-head 比较的公开报告。

### 5.2 模型与结果

| 指标 | 数值 |
|------|------|
| 总参数 | 200B (MoE) |
| 激活参数 | 20B |
| AIME 2024 | 86.7 |
| Codeforces | 55.0 |
| GPQA | 77.3 |

### 5.3 核心发现

**发现 1：VAPO 稳定优于 DAPO（+10 分）**

> 报告 Table 2: 在 AIME 等多个 benchmark 上，VAPO 一致性领先 DAPO 约 10 分。

这验证了 VAPO 论文的核心主张："Value-Free 在短任务上够用，但在长 CoT 推理中，修好的 Value-Based 更强"。VAPO 的三大 Critic 修复技术（Value Pretraining, Decoupled GAE, Length-Adaptive GAE）在 200B MoE 上同样有效。

**发现 2：RFT 伤害后续 RL（非常反直觉）**

> 报告 Section 4.3: 先做 Rejection Sampling Fine-tuning 再做 RL，效果**不如直接从 Cold Start SFT 进入 RL**。

传统思路认为"先用 RFT 提升起点 → 再用 RL 进一步优化"应该更好。但数据表明：

| Pipeline | AIME 2024 |
|----------|-----------|
| Cold Start SFT → RL | **更高** |
| Cold Start SFT → RFT → RL | 更低 |

**报告给出的分析**：
1. RFT 让模型输出分布变窄（只保留"正确"模式），降低了 RL 阶段的探索多样性
2. RL 需要模型在"错误"中学习——如果模型已经很少犯错，RL 信号变弱
3. RFT 可能引入 "false positive" 模式——推理路径有缺陷但答案碰巧正确的回答

**实践建议**: 如果最终目标是做 RL，考虑跳过 RFT 阶段。

**发现 3：DAPO 在大规模上依然有效**

作为完全开源方案，DAPO 在 200B MoE 上验证了其四大改进（Clip-Higher, Dynamic Sampling, Token-Level Loss, Overlong Reward Shaping）的实际价值。

## 六、LLaMA 3.1 / 4 — 从 DPO 到 Online RL 的转向

> **LLaMA 3.1 报告**: *The Llama 3 Herd of Models*
> **LLaMA 3.1 arXiv**: [2407.21783](https://arxiv.org/abs/2407.21783) (2024.07)
>
> **LLaMA 4 博客**: [ai.meta.com/blog/llama-4-multimodal-intelligence/](https://ai.meta.com/blog/llama-4-multimodal-intelligence/) (2025.04)
>
> **核心叙事**: Meta 从 LLaMA 3.1 到 LLaMA 4 的后训练策略发生了根本性转向

### 6.1 LLaMA 3.1：DPO 路线的巅峰

LLaMA 3.1 报告（arXiv:2407.21783）是目前最详尽的工业级 Post-Training 技术报告。其核心特点是 **6 轮迭代训练**：

```
每一轮:
  1. 训练/更新 Reward Model
  2. SFT + Rejection Sampling (用 RM 筛选最佳回答)
  3. DPO 对齐
重复 6 轮，每轮的 RM 和数据都在改进
```

**核心发现（有争议）：DPO > PPO**

LLaMA 3.1 团队在他们的实验设置下得出结论：**DPO 在一般对齐任务上优于 PPO**。但需注意：

| 因素 | LLaMA 3.1（DPO 优） | DeepSeek-R1（RL 优） |
|------|---------------------|---------------------|
| 任务类型 | 通用对齐（helpful, safe） | 复杂数学推理 |
| 回答长度 | 相对较短 | 长 CoT（数千 token） |
| 评估方式 | 人类偏好评估 | 数学答案正确性 |

**10 条关键训练经验**（报告 Section 4）：

1. **Format Token Masking**: DPO 中 mask 格式 token，只优化内容
2. **NLL 正则化 ($\lambda=0.2$)**: 防止 DPO 只学会降低坏回答概率而不提升好回答
3. **Rejection Sampling > 直接 SFT**: 用 RM 筛选模型自身生成的最佳回答效果更好
4. **安全性需要独立阶段**: 有用性和安全性的优化目标存在张力
5. **迭代训练是核心**: 6 轮迭代中每一轮都在改进，单轮远不够

### 6.2 LLaMA 4：转向 Online RL

LLaMA 4 的后训练发生了**根本性转向**，从 DPO 重心转移到 Online RL：

```
LLaMA 3.1: Heavy SFT → 6轮迭代 DPO
LLaMA 4:   Lightweight SFT → Online RL → Lightweight DPO
```

**转向的动机**（根据公开信息推断）：

1. **DeepSeek-R1 的成功冲击**: 纯 RL 涌现推理能力的结果让 Meta 重新评估在线 RL 的价值
2. **过重的 SFT/DPO 限制 RL 探索空间**: LLaMA 4 团队发现过多的离线优化反而限制了模型的探索能力
3. **Llama-as-judge 数据过滤**: 使用模型自身判断删除 50%+ "简单"训练数据，保留真正有梯度信号的样本

**架构变化**: LLaMA 4 采用 MoE 架构（Scout: 17B×16 专家, Maverick: 17B×128 专家），与 DeepSeek-V3 的 MoE 趋势一致。

## 七、GLM-5 — 异步 Agent RL 与工程教训

> **报告**: *GLM-5: from Vibe Coding to Agentic Engineering*
> **arXiv**: [2602.15763](https://arxiv.org/abs/2602.15763) (2026.02)
>
> **前代报告**: *ChatGLM: A Family of Large Language Models from GLM-130B to GLM-4 All Tools*
> **前代 arXiv**: [2406.12793](https://arxiv.org/abs/2406.12793)
>
> **核心主张**: 异步 RL 基础设施解耦生成与训练，实现 Agent RL 的高效训练

### 7.1 动机分析

GLM-5 面对的核心挑战是：**如何在端到端软件工程等长程交互任务上进行 RL 训练？** 传统的同步 RL（生成一批 → 计算奖励 → 更新参数 → 生成下一批）在 Agent 场景中效率极低，因为每一步工具调用都需要等待环境响应。

### 7.2 Pipeline：五阶段

```
GLM-5 Base
    │
    ▼ 阶段 1: SFT — 通用指令跟随
    ▼ 阶段 2: Reasoning RL — 数学/代码/逻辑
    ▼ 阶段 3: Agentic RL — 工具使用/多轮交互 ← 核心创新
    ▼ 阶段 4: General RL — 全场景对齐
    ▼ 阶段 5: 跨阶段蒸馏 — 整合各阶段最优能力
    │
    ▼
GLM-5 Chat (SWE-bench SOTA)
```

### 7.3 核心技术创新

**异步 RL 基础设施**

报告提出将 RL 的生成（rollout）和训练（update）**解耦为异步流水线**：

- **生成进程**: 持续与环境交互，将完成的轨迹放入缓冲区
- **训练进程**: 从缓冲区取轨迹，计算梯度，更新参数
- 两个进程异步运行，避免了同步等待的瓶颈

这对 Agent RL 至关重要——Agent 任务中每步工具调用可能需要数秒到数分钟（如执行代码、调用 API），同步等待会让 GPU 利用率极低。

**TITO Gateway (Think-In, Think-Out)**

模型可以在输出中嵌入"思考段落"，这些思考段落在最终输出时被过滤掉。类似于 Qwen3 的 Thinking Mode，但实现为一个显式的 gateway 机制。

**跨阶段蒸馏（阶段 5）**

解决 RL 各阶段之间的**能力遗忘**问题：

- Reasoning RL 提升推理但可能损害对话
- Agentic RL 提升工具使用但可能损害推理
- 跨阶段蒸馏将每个阶段的"最优版本"作为教师，综合蒸馏到最终模型

### 7.4 关键工程教训

**非确定性 CUDA Top-K 的致命问题**（报告 Section 5.3）

> "CUDA 的 top-k 操作在某些情况下是非确定性的（nondeterministic），这会导致 RL 训练在几步之内崩溃。"

**根因分析**: RL 训练中，IS 比率 $r_{i,t} = \pi_\theta / \pi_{\theta_{\text{old}}}$ 需要精确的概率分布。如果 top-k 采样在不同前向传播中给出不同结果（GPU 浮点运算的非确定性），$r_{i,t}$ 出现异常值 → 梯度爆炸 → 训练崩溃。

**解决方案**: 强制使用确定性的 top-k 实现（牺牲一些性能），或在 RL 训练阶段避免 top-k 采样。

这个发现对所有做 MoE + RL 的团队都有警示意义。

## 八、Gemini 2.5 — 可验证奖励与思考预算

> **技术报告**: [Gemini 2.5 Report (PDF)](https://storage.googleapis.com/deepmind-media/gemini/gemini_v2_5_report.pdf) (2025.03)
> **博客**: [Google DeepMind Blog](https://blog.google/technology/google-deepmind/gemini-model-thinking-updates-march-2025/)
>
> **相关论文**:
> - BOND: *Best-of-N Distillation*
> - WARM: *Weight Averaged Reward Models* (arXiv:2401.12187)
> - WARP: *Weight Averaged Rewarded Policies*
> - Gemma 3 Technical Report: [arXiv:2503.19786](https://arxiv.org/abs/2503.19786)
>
> **核心主张**: 重度使用可验证奖励 + 基于模型的生成式奖励，实现 LMArena Elo +122 分

### 8.1 动机分析

Google 的 Post-Training 哲学与其他厂商有显著差异。他们的核心思路不是"找到最好的 RL 超参数"，而是"**训练多个模型，然后在权重空间中平均**"——这是一种更稳健、更可复现的路线。

### 8.2 Gemini 2.5 的后训练策略

报告透露了几个关键信息：

1. **大幅增加 RL 计算量**: 相比前代，Gemini 2.5 在 RL 阶段的算力投入显著增加
2. **重点使用可验证奖励**: 与 RLVR 范式一致，在有确定答案的任务上使用规则奖励
3. **基于模型的生成式奖励**: 对开放性任务，使用 LLM-as-Judge 生成奖励
4. **可控思考预算**: Gemini 2.5 Flash 支持用户指定思考预算（低/中/高），模型按预算分配推理计算

**LMArena Elo 提升**:
- Gemini 2.5 Pro: +122 分
- Gemini 2.5 Flash: +111 分

### 8.3 权重平均哲学

Google 的三个代表性方法，构成了一个独特的技术路线：

| 方法 | 做法 | 核心思想 |
|------|------|---------|
| **BOND** | 对 prompt 采样 N 个回答，用 RM 选最佳，做 SFT 蒸馏 | 类似 VAPO 的 Positive-Example LM Loss |
| **WARM** | 训练多个 RM，在**权重空间**平均 | 权重平均的 RM 比单个更鲁棒 |
| **WARP** | 多轮 RL（不同种子），在**权重空间**平均策略模型 | 多次 RL + 权重平均 > 单次精调 |

**为什么权重平均有效？**

不同训练轨迹收敛到损失曲面的不同局部最优，这些局部最优各有优缺点。权重平均在这些局部最优之间找一个"中间点"，这个中间点往往比任何单个最优都好（前提是这些模型在同一个损失盆地中）。

### 8.4 RM 设计特点

| 特点 | 描述 | 与其他厂商的对比 |
|------|------|----------------|
| RM ~10× 策略模型 | RM 参数量远大于策略模型 | DeepSeek/Qwen 用同等或更小的 RM |
| 权重平均 RM | 多个 RM 在权重空间平均 | 其他厂商多用单一 RM |
| 推理时多 RM 投票 | 部署时用多个 RM 集成打分 | 更高的推理成本换取更可靠的奖励 |

## 十、跨模型训练经验总结

综合以上 2025-2026 年所有模型的 Post-Training 实践，提炼出以下关键经验：

### 经验 1：Pipeline 正在收敛

尽管各家实现不同，大框架在 2025 年趋同：

```
Lightweight SFT → Reasoning RL (RLVR) → [Agentic RL] → General RL (RLHF)
```

**证据**:
- DeepSeek-R1 (arXiv:2501.12948): Cold Start SFT → GRPO → Rejection Sampling SFT → All-Scenario RL
- Qwen3 (arXiv:2505.09388): Long-CoT SFT → GRPO → Thinking Mode Fusion SFT → GSPO
- LLaMA 4 (blog): Lightweight SFT → Online RL → Lightweight DPO
- GLM-5 (arXiv:2602.15763): SFT → Reasoning RL → Agentic RL → General RL → Cross-Stage Distillation

### 经验 2：Online RL 在推理任务上不可替代

| 方法 | 通用对齐 | 复杂推理 | 证据 |
|------|---------|---------|------|
| DPO | 好 | 一般 | LLaMA 3.1 (arXiv:2407.21783) |
| Online RL (GRPO/DAPO) | 好 | 强 | DeepSeek-R1, Qwen3 |
| Online RL + Value (VAPO) | 好 | **最强** | Seed1.5 (arXiv:2504.13914): VAPO 稳定 +10 分 |

### 经验 3：数据质量 > 数据数量 > 模型规模

| 证据 | 来源 |
|------|------|
| 3,995 个 query + 170 步 RL 提升 15 分 | Qwen3 (arXiv:2505.09388) |
| QwQ-32B 性能匹配 DeepSeek-R1 671B MoE | Qwen 系列 |
| R1 蒸馏仅 5K H800 hours | DeepSeek-V3 (arXiv:2412.19437) |

### 经验 4：RL 的作用是"释放"而非"教授"

多个模型的经验表明，基座模型在预训练中已学到推理的"种子"，RL 的作用是选择性强化。

- R1-Zero 的 "aha moment" 是涌现的，不是教会的 (arXiv:2501.12948)
- Qwen3 仅 170 步 RL 就提升 15 分 (arXiv:2505.09388)
- SimpleRL-Zoo (arXiv:2503.18892) 在 10 种不同基座模型上复现了涌现行为

### 经验 5：RFT 不一定是好的 RL 预处理

Seed1.5-Thinking (arXiv:2504.13914) 的发现：RFT 可能限制后续 RL 的探索空间。如果最终目标是做 RL，考虑直接从 Cold Start SFT 进入 RL。

### 经验 6：MoE 模型的 RL 训练有特殊挑战

| 挑战 | 表现 | 解决方案 | 来源 |
|------|------|----------|------|
| 专家路由不稳定 | token-level IS ratio 剧烈波动 | GSPO（序列级） | arXiv:2507.18071 |
| Loss spike | 训练中偶发损失尖峰 | MuonClip 优化器 | Kimi K2 (arXiv:2507.20534) |
| 非确定性计算 | CUDA top-k 导致 RL 崩溃 | 强制确定性实现 | GLM-5 (arXiv:2602.15763) |

### 经验 7：蒸馏是高效获取推理能力的捷径

| 方法 | GPU 成本 | 效果 | 来源 |
|------|---------|------|------|
| 从头 RL | 100% | 最强 | DeepSeek-R1 |
| 强到弱蒸馏 | ~10% | 接近教师水平 | Qwen3 (arXiv:2505.09388) |
| R1 蒸馏 pipeline | ~5% | 中等水平 | DeepSeek-V3 (arXiv:2412.19437) |

### 经验 8：训练阶段之间存在能力遗忘

每个 RL 阶段优化特定能力时会损害其他能力。解决方案：
- **混合训练数据**: PPO-ptx 思路 (InstructGPT, arXiv:2203.02155)
- **跨阶段蒸馏**: GLM-5 的第五阶段 (arXiv:2602.15763)
- **迭代训练**: LLaMA 3.1 的 6 轮迭代 (arXiv:2407.21783)

### 经验 9：安全性和有用性需要分阶段优化

| 模型 | 安全性处理方式 | 来源 |
|------|--------------|------|
| LLaMA 3.1 | 独立安全性 RL 阶段 | arXiv:2407.21783 |
| Qwen2.5 | 6 维 RM 中安全性独立维度 | arXiv:2412.15115 |
| DeepSeek-R1 | All-Scenario RL 包含安全性 | arXiv:2501.12948 |
| o3-mini | Deliberative Alignment | OpenAI blog |
